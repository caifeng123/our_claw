# TikTok Shop 东南亚电商商品标题生成器

## 概述

本 Skill 帮助跨境电商卖家为 Sanrio（三丽鸥）及类似 IP 周边小物件生成适合东南亚五国 TikTok Shop 的本地化商品标题。结合产品特征分析和当地营销节日，产出高质量、SEO 友好的商品标题。

## 双模式运行

本 Skill 有两种运行模式，**自动判断**，无需用户手动选择：

### 模式 A：完整分析（用户发送了产品图片）

```
产品图片 → 图片分析 → 保存产品特征 → 运行节日脚本 → 生成标题
```

### 模式 B：刷新更新（用户没有发送图片）

```
运行节日脚本（含差异检测）→ 读取已保存产品特征 → 仅更新有变化的市场标题
```

**判断规则：**
- 用户消息中包含产品图片 → 模式 A
- 用户消息中没有图片，但要求刷新/更新标题 → 模式 B
- 用户明确说"全部刷新" → 模式 B，但加 `--force` 参数跳过差异检测
- 模式 B 运行时如果找不到已保存的产品特征文件 → 提示用户先发送产品图片

## 步骤 1：产品图片分析（仅模式 A）

用户通常会发送 3-4 张产品图（主图、细节图、场景图等）。产品范围聚焦于 Sanrio（三丽鸥）及类似 IP 周边小物件：摆件、手表、钥匙扣、文具、手机壳、毛绒公仔等。

对每张图进行仔细观察，提取以下 **11 个维度** 的信息，尽可能详细：

| 维度 | 说明 | 示例 |
|---|---|---|
| 产品类型 | 具体品类 | 树脂摆件、石英手表、亚克力钥匙扣 |
| IP角色识别 | 识别具体角色及系列 | Kuromi（酷洛米）、My Melody（美乐蒂）、限定联名系列 |
| 核心功能 | 产品的实际用途 | 桌面装饰、计时、收纳、挂饰 |
| 材质判断 | 从图片推断材质 | PVC、树脂、合金、硅胶、亚克力 |
| 尺寸估计 | 相对大小判断 | 迷你（<5cm）、中等（5-15cm）、大型（>15cm） |
| 卖点提炼 | 3-5 个核心卖点 | 限定款、可动关节、可收集全套、精致细节 |
| 适用人群 | 目标消费者画像 | 女学生、上班族、IP收藏者、送礼人群 |
| 使用场景 | 实际使用场景 | 桌面装饰、日常佩戴、送礼、拍照道具、车载摆件 |
| 价格带定位 | 预估市场定位 | 低价引流（<$5）、中端（$5-15）、限定溢价（>$15） |
| 风格标签 | 审美风格关键词 | 可爱、少女心、治愈系、童趣、Y2K、甜酷 |
| 配色分析 | 主色调及特征 | 粉紫配色、马卡龙色系、黑粉撞色、季节限定配色 |

### 产品识别与管理规则

- **自动命名**：首次分析时，根据图片识别结果自动生成文件名 `{角色名}_{产品类型}_{序号}.json`
- **增量管理**：每次发图只处理本次图片对应的产品，不影响已有产品文件
- **覆盖更新**：对已存在的产品重新发图 → 匹配到同名文件 → 覆盖内容，刷新 `updated_at`
- **模式 B 遍历**：刷新标题时，自动扫描 `products/` 目录下所有 JSON 文件，逐个生成标题
- **指定产品**：用户可以指定某个产品刷新（如"刷新手表的标题"），模糊匹配文件名

## 步骤 2：节日获取（脚本确定性筛选）

节日数据完全来自本地 `holidays_YYYY.json`，不联网搜索。由 Python 脚本精确完成时间窗口筛选和差异检测。

### 运行节日引擎脚本

**脚本路径：** `scripts/holiday_engine.py`（位于本 Skill 目录下）

**调用命令：**
```bash
python3 <skill_path>/scripts/holiday_engine.py \
  --data-dir <project_root>/data/project-name \
  --days 30
```

**模式 B 强制刷新时加 `--force`：**
```bash
python3 <skill_path>/scripts/holiday_engine.py \
  --data-dir <project_root>/data/project-name \
  --days 30 \
  --force
```

**脚本做的事情：**

1. **精确计算时间窗口**：`today → today + 30天`，Python `datetime` 零误差
2. **硬筛节日**：从 `holidays_YYYY.json` 中按日期过滤，窗口外直接丢弃
3. **差异检测**：对比 `cache/last_holidays.json`，按市场粒度输出变化
4. **输出结构化 JSON**：含每个市场的节日列表 + `holiday_keywords`（本地语言关键词）

**脚本输出结构：**
```json
{
  "window_start": "2026-04-12",
  "window_end": "2026-05-12",
  "markets": {
    "SG": {
      "holidays": [...],
      "holiday_count": 4,
      "holiday_keywords": ["Labour Day", "May Day", "Mother's Day", "Gift for Mom", "Payday Sale", "5.5 Sale"],
      "changed": true,
      "diff_summary": "新增: Mother's Day; 移除: Good Friday"
    }
  }
}
```

直接使用脚本输出，无需额外处理。脚本输出即最终节日列表。

### 节日数据文件

节日数据存放在 `<skill_dir>/data/holidays_YYYY.json`（随 Skill 一起入 git 分发）。

**数据分三层：**

| 类型 | 示例 | 日期特点 |
|---|---|---|
| 固定节日 | 劳动节 5/1、国庆日 | 每年相同，一次性写入 |
| 浮动宗教节 | 开斋节、卫塞节、春节 | 同年内确定，每年初需更新 |
| 电商造节 | 5.5/6.6/Payday Sale | 固定规则，一次性写入 |

**年度维护**：每年 1 月初，手动更新 `<skill_dir>/data/holidays_YYYY.json` 中浮动节日的日期，提交到 git。

### 节日优先级

| 优先级 | 类型 | 示例 |
|---|---|---|
| **P0** | 国家法定假日 / 宗教大节 | 开斋节、泼水节、越南统一日 |
| **P1** | 国际通用节日 | 母亲节、情人节、儿童节 |
| **P2** | 电商平台造节 | 5.5大促、Payday Sale |

## 步骤 3：生成 TikTok Shop 商品标题

### 生成规格

| 项目 | 要求 |
|---|---|
| 每个市场 | 生成 **3 条**候选标题 |
| 字符限制 | 每条 ≤ **255** 字符，**尽可能写满** |
| 格式 | **纯文字**，不使用 emoji |
| 翻译 | 每条标题附**中文翻译** |
| 批量处理 | 如果有多个已保存的产品，对每个产品都生成标题 |

### 各市场语言要求

| 市场 | 标题语言 | 说明 |
|---|---|---|
| SG | **英语** | 新加坡通用英语 |
| MY | **马来语 + 英语混合** | 马来西亚电商习惯双语混合，IP名/品类词常用英语，不出现商品品牌 |
| TH | **泰语** | 完整泰语，仅 IP 名称保留英文，不出现商品品牌 |
| PH | **英语**（可混少量 Tagalog） | 菲律宾 TikTok Shop 主流用英语 |
| VN | **越南语** | 完整越南语，仅 IP 名称保留英文，不出现商品品牌 |

### 核心策略：全部节日关键词塞入标题

这是本 Skill 最重要的标题生成策略——**把脚本输出的 `holiday_keywords` 中的所有关键词都塞进 255 字符的标题里**。

原因：TikTok Shop 的搜索算法是关键词匹配，标题里包含的节日关键词越多，在不同节日期间被搜索到的概率就越高。比如一条标题同时包含 "Mother's Day" + "5.5 Sale" + "Labour Day" + "Payday Sale"，那这四个节日期间都能被搜到。

**标题关键词优先级（从高到低）：**

1. **IP 名称 + 品类词**（必须在最前面，搜索结果只显示前半段）
2. **全部 P0 节日关键词**（本地语言）
3. **全部 P1 节日关键词**（本地语言）
4. **全部 P2 节日关键词**（本地语言）
5. **产品属性词**（材质、风格、适用人群等）
6. **补充搜索词**（到 255 字符上限）

### 标题写作原则

1. **前置核心信息**：IP 名称 + 品类词放在最前面
2. **节日关键词全覆盖**：读取脚本输出的 `holiday_keywords` 列表，尽可能全部嵌入标题
3. **本地化表达**：使用当地消费者的习惯用语，不是简单翻译
4. **255 字符用满**：每一个字符都有价值，尽量接近上限
5. **可读性兜底**：虽然要堆关键词，但标题仍需基本可读，不能是纯关键词罗列
6. **禁止出现商品品牌名**：标题中不得出现 MOETCH、TOPTOY、茉趣、阿里鱼等商品品牌/经销商名称。IP 角色名（Hello Kitty、Kuromi、My Melody 等）和版权方名（Sanrio）可以保留。
7. **禁止出现授权**：标题中不得出现授权等字样，最多表明正版即可。
8. **禁止出现随机盲盒字样**：标题中不得出现盲盒/神秘商品等字样。

### 标题结构参考

```
[IP名称] [产品类型] [核心卖点] [P0节日关键词] [P1节日关键词] [P2节日关键词] [材质/属性] [适用场景] [补充词...]
```

## 差异检测（由脚本完成）

差异检测逻辑已固化在 `scripts/holiday_engine.py` 中，模型不需要自行判断。

### 对比规则

脚本对比的是**上次缓存的30天窗口节日列表** vs **本次计算的30天窗口节日列表**，按 `(节日名, 日期)` 元组做集合差异：

| 情况 | 触发条件 | `changed` | 行为 |
|---|---|---|---|
| **新增节日** | 窗口滑动，新节日进入30天范围 | `true` | 重新生成该市场标题 |
| **移除节日** | 窗口滑动，旧节日滑出30天范围 | `true` | 重新生成（关键词需更新） |
| **无变化** | 两次运行间窗口边界未跨过任何节日 | `false` | 跳过（脚本已从 stdout 过滤） |
| **首次运行** | 无缓存文件 | `true` | 全部生成 |
| **强制刷新** | 用户说"全部刷新"，加 `--force` | `true` | 忽略缓存，全部重新生成 |

### 市场粒度独立检测

5个市场各自独立判断。例如窗口滑动导致 TH 的 Songkran 滑出但其他4国无变化，则只重新生成 TH 的标题，其余4国跳过不输出。

## 注意事项

- 泰语、越南语等语言的字符计数方式与英语不同，生成标题时要确认实际字符数不超过 255
- 马来西亚市场的双语混合标题，注意马来语和英语的自然衔接
- 产品特征文件是长期资产，分析要尽量准确详细，后续所有标题生成都依赖它
- 每年 1 月初需更新 `<skill_dir>/data/holidays_YYYY.json` 中浮动节日的日期并提交 git
