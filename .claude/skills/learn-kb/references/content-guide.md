# 内容总结与文档生成规范

生成飞书知识库文档时参考此规范。目标：让每篇总结成为「未来的自己能快速回忆并使用」的高质量笔记。

## 文档模板

```markdown
> [!NOTE]
> 一句话核心观点（或：这个项目/工具是什么，一句话说清楚）

## 内容概述

2-3 句话说明来源、核心主题、为什么值得保存。
不要写"本文介绍了..."这种废话开头，直接说重点。

## 核心观点

按内容类型组织（见下方"按类型的总结策略"）。
用自己的语言重组信息，不是摘抄原文。
每个观点配一句"为什么这很重要"或"这意味着什么"。

## 值得记住的启发

提炼 2-5 条对读者有价值的 takeaway：
- 改变了我对某件事的认知
- 可以直接用到工作中的方法论
- 值得进一步深挖的方向

## 对项目的帮助

这是知识库文档最有价值的章节。不是泛泛说"有参考意义"，而是**具体到项目的模块、代码、流程**，说清楚这篇内容能带来什么帮助。

要回答以下问题（按适用性选写，不必每条都有）：

### 可以直接借鉴什么？
- 哪个思路、模式、方案可以直接搬到项目里？
- 对应项目中的哪个模块 / 哪段代码 / 哪个流程？
- 搬过来需要做什么适配？

### 可以优化项目的哪些地方？
- 读完之后，项目里有哪些做法可以改进？
- 改进的方向是什么？预期收益是什么？
- 优先级如何？（立刻可做 / 后续规划 / 仅作储备）

### 有什么值得警惕的？
- 文中提到的坑、限制、踩过的雷，项目里是否也存在？
- 需要提前规避什么？

写法要求：
- **必须基于 Design.md 的项目锚点**做匹配，不要凭空猜测。先明确命中了哪个模块/技术栈/设计难题，再展开分析
- **具体到模块级别**——"context-builder 的增量压缩策略可以借鉴文中的 ZZ 机制"，而不是"这个很有参考意义"
- **诚实评估**——关联度低就说低，不硬凑。"这篇讲的是移动端动画，跟 my_claw 的 Agent 架构没有直接关联。不过'感知性能 vs 实际性能'的思路，在优化 streaming-card-renderer 的体感速度时可以借鉴。"

好的例子：

> ### 可以直接借鉴什么？
> 文中的「指数退避 + 断路器」重试模式，可以直接用于 `claude-engine.ts` 调用 Claude SDK 和 `feishu-service.ts` 调用飞书 Open API 的场景。
> 当前 Agent 引擎遇到 API 报错直接抛异常退出，没有重试。按照文中方案改造后，能扛住 Claude API 偶发的 529/500 和飞书 API 的 429 错误。
>
> ### 可以优化项目的哪些地方？
> 1. `memory-db.ts` 的驱逐策略——文中提到的"时间窗口 + 访问频次"衰减算法比当前的线性重要度衰减更精准，可以减少误淘汰高价值记忆
> 2. `context-builder.ts` 的压缩策略——文中提到的"分层摘要"思路（近期对话保留细节、远期对话只保留结论），比当前的一刀切压缩保留更多有效信息
>
> ### 有什么值得警惕的？
> 文中提到 Claude API 在长上下文 + 工具调用并发时有隐性的延迟尖刺（不在官方文档里），vision-guard 的 Sub-Agent 调用场景需要注意。

## 可直接复用的内容

代码片段、命令、清单、模板、配置等。
如果没有可复用的具体内容，省略此章节。

## 一句话总结

> [!TIP]
> 一句话总结全文核心价值。

---

## 原始链接

- 原文：[标题](URL)
- 抓取时间：YYYY-MM-DD
```

## 按类型的总结策略

### 短笔记/帖子（小红书、社交分享）

特点：内容简短，可能不完整，碎片化信息。

策略：
- 提取核心要点，不假装有完整原文
- 关注可操作建议（"做什么"而不是"说了什么"）
- 如果是经验分享，提炼成清单
- 补充背景信息帮助理解

### 长文章/博客

特点：有完整论证结构，有观点有论据。

策略：
- 提炼论点层次：主论点 → 分论点 → 关键论据
- 关注论证逻辑而非细节堆砌
- 识别作者的独到见解 vs 常识性内容
- 如果有数据支撑，保留关键数字

### 技术/产品文档

特点：API 文档、工程博客、Release Note、设计文档。

策略：
- 关注「设计选择」——为什么选 A 而不是 B
- 提取工作流和关键步骤
- 记录约束条件、已知限制、注意事项
- 保留可复用的代码示例和命令

### GitHub 仓库

特点：README 为主，可能有 docs 目录。

策略：
- 一句话说清项目定位（解决什么问题）
- 核心功能列表（不是功能的流水账，而是"最有价值的几个能力"）
- 技术栈和架构概览
- 快速上手步骤（如果有）
- 与同类项目的差异点

## 写作原则

1. **为未来的自己写**：6 个月后再看这篇总结，能否在 30 秒内回忆起为什么保存它？
2. **结构化重组**：不是摘抄原文段落，而是用自己的语言重新组织
3. **标注不确定性**：抓取不完整时明确说明缺失部分，绝不编造
4. **区分来源**：原文观点用"作者认为..."，自己的补充用"补充："或"注："
5. **保留可操作内容**：代码、命令、清单、模板等直接复用的内容原样保留
6. **控制篇幅**：短笔记总结 200-500 字，长文章 500-1500 字，技术文档按需

## 标签生成规范

目标：让标签成为未来检索的入口。好的标签体系能让你通过多个维度找到同一篇内容。

### 维度覆盖

| 维度 | 示例 | 说明 |
|------|------|------|
| 内容性质 | 技术解析、技术实践、产品思考、行业洞察、工具推荐、学习笔记 | 这篇内容"是什么" |
| 技术栈 | Python、React、Docker、K8s、LLM、PostgreSQL | 涉及的具体技术 |
| 领域 | 前端、后端、AI、数据工程、DevOps、安全、移动端 | 所属大领域 |
| 主题 | 架构设计、性能优化、Prompt Engineering、开源项目 | 讨论的核心话题 |
| 场景 | 面试、入门教程、生产实践、最佳实践、源码分析、踩坑记录 | 适用什么场景 |

### 生成原则

- **5-15 个标签**，宁多勿少
- **粒度适中**：太粗（"技术"）无用，太细（"Python 3.11.4"）也无用
- **用中文**，简洁统一（"大模型"而非"大型语言模型"）
- **可检索**：想象自己未来会用什么关键词搜索这篇内容


## 关联度评判标准

基于 `Design.md` 中的项目锚点逐条匹配：

| 关联度 | 判定条件 | 示例 |
|--------|----------|------|
| **极高** | 直接解决某个核心模块的已知问题 | "文中的流式渲染优化方案可以直接用于 streaming-card-renderer" |
| **高** | 命中核心模块 + 有明确可搬方案 | "Agent 工具调度的新模式可以优化 tool-manager 的注册机制" |
| **中** | 命中技术栈但需要较大适配 | "TypeScript 的错误处理最佳实践，需要逐模块改造" |
| **低** | 间接相关，仅作知识储备 | "其他 Agent 框架的设计理念，暂时用不上但值得了解" |
| **极低** | 与项目锚点无交集 | "移动端 UI 框架选型" |

无 `Design.md` 时默认判「中」，不做极端判断。

## 按来源类型调整「对项目的帮助」深度

| 来源类型 | 分析深度 | 理由 |
|----------|----------|------|
| 短笔记/小红书 | **轻**：1-2 句话，不强制三个子章节 | 内容本身碎片化，硬分析反而注水 |
| 长文章/博客 | **标准**：完整三个子章节 | 有足够内容支撑深度分析 |
| GitHub 仓库 | **标准 + 重点写「可以直接借鉴什么」** | 最容易产出具体方案 |
| 技术文档/API | **重**：保留代码示例 + 具体适配步骤 | 最有实操价值 |
